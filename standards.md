## Industry Standards & Specifications

### ISO Standards

**ISO 9001:2015 / ISO 9001:2026 (Quality Management Systems)**
- URL: https://www.iso.org/standard/62085.html
- Defines documented information requirements (Clause 7.5) covering how documents are created, reviewed, approved, and updated. Directly relevant to how SOW documents must be version-controlled, identifiable, traceable, and accessible. The forthcoming ISO 9001:2026 revision reinforces alignment with digital workflows and quality culture.

**ISO 19475:2021 — Document Management: Minimum Requirements for the Storage of Documents**
- URL: https://www.iso.org/standard/76865.html
- Specifies minimum requirements for the secure, long-term storage of business documents. Applies to archiving accepted SOWs and maintaining audit trails for contract disputes or audits.

**ISO 4669-1:2023 — Document Management: Information Classification, Marking and Handling**
- URL: https://www.iso.org/standard/80233.html
- Covers classification and handling of information in documents. Relevant to tagging SOW documents with sensitivity levels (e.g. confidential pricing, proprietary scope) and enforcing access controls.

**ISO 32000-2:2017 — Document Management: Portable Document Format (PDF 2.0)**
- URL: https://www.iso.org/standard/63534.html
- Defines the PDF 2.0 file format specification. SOW generators that export to PDF for delivery and e-signature must conform to this standard. PAdES-compliant PDF signatures build on this foundation.

**ISO 27001 — Information Security Management**
- URL: https://www.iso.org/standard/27001
- Annex A 5.31 covers legal, statutory, regulatory and contractual requirements. Relevant to ensuring that SOW content (including commercially sensitive pricing and client data) is stored, transmitted, and accessed in compliance with security obligations.

### W3C & IETF Standards

**RFC 9457 — Problem Details for HTTP APIs** (obsoletes RFC 7807)
- URL: https://www.rfc-editor.org/rfc/rfc9457.html
- Defines a standard JSON format for machine-readable error responses from HTTP APIs, including `type`, `title`, `status`, `detail`, and `instance` fields. A SOW generator's REST API should use this format for consistent error communication with integrating clients.

**RFC 8288 — Web Linking**
- URL: https://www.rfc-editor.org/rfc/rfc8288.html
- Defines the `Link` header for expressing relationships between web resources. Relevant to linking SOW document versions, attachments, and approval workflow states via hypermedia.

**RFC 6749 — The OAuth 2.0 Authorization Framework**
- URL: https://datatracker.ietf.org/doc/html/rfc6749
- Industry-standard framework for delegated authorisation. SOW generators that integrate with CRMs (Salesforce, HubSpot), cloud storage (Google Drive, SharePoint), and e-signature platforms (Docusign, Dropbox Sign) must implement OAuth 2.0 for secure API access.

**RFC 7519 — JSON Web Token (JWT)**
- URL: https://datatracker.ietf.org/doc/html/rfc7519
- Defines compact, URL-safe tokens used for authentication and claims transmission. Relevant to API-to-API authentication in multi-system SOW workflows (CRM → SOW generator → e-signature platform).

**W3C Verifiable Credentials (VC) Data Model**
- URL: https://www.w3.org/TR/vc-data-model/
- Provides a standard for expressing credentials in a machine-verifiable way. Emerging relevance as EU Digital Identity Wallet (eIDAS 2.0) deployments make it possible for signatories to prove identity using verifiable credentials, strengthening the legal standing of signed SOWs.

### Data Model & API Specifications

**OpenAPI Specification 3.1 / 3.2**
- URL: https://spec.openapis.org/oas/v3.2.0.html
- The industry-standard format for describing REST APIs in a machine-readable, language-agnostic way. A SOW generator API should publish an OpenAPI document to enable automated code generation, interactive documentation, mock servers, and contract testing by integrating clients.

**JSON Schema (Draft 2020-12)**
- URL: https://json-schema.org/
- Vocabulary for annotating and validating JSON documents. Used to define and validate the structured data model of a SOW (sections, deliverables, milestones, payment terms, assumptions) passed between systems or generated by AI models.

**CloudEvents 1.0 — CNCF Event Specification**
- URL: https://cloudevents.io/
- A CNCF-graduated, vendor-neutral specification for describing event data in a common format across services, platforms, and systems. Relevant to notifying downstream systems (billing, project management, CRM) when SOW lifecycle events occur: created, approved, signed, revised, or rejected.

### Security & Authentication Standards

**OWASP API Security Top 10**
- URL: https://owasp.org/www-project-api-security/
- Defines the most critical API security risks. A SOW generator API handling commercially sensitive pricing, client data, and legal clauses must address broken object-level authorisation, excessive data exposure, and injection vulnerabilities.

**OWASP REST Security Cheat Sheet**
- URL: https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html
- Practical guidance for securing REST APIs, including input validation (JSON Schema), proper HTTP status codes, and authentication patterns. Directly applicable to the SOW generator's external-facing API.

**eIDAS Regulation (EU) No 910/2014 / eIDAS 2.0**
- URL: https://www.onespan.com/topics/eidas
- EU regulation governing electronic identification and trust services, including the legal validity of electronic signatures on contracts. SOWs signed digitally in the EU must meet at minimum a Simple Electronic Signature (SES), with enterprise use cases potentially requiring Advanced (AES) or Qualified (QES) signatures. eIDAS 2.0 / EU Digital Identity Wallet further strengthens interoperability for 2026+.

**ETSI EN 319 142 — PAdES (PDF Advanced Electronic Signatures)**
- URL: https://www.etsi.org/deliver/etsi_en/319100_319199/31914201/
- ETSI standard for embedding advanced electronic signatures in PDF documents. Relevant to SOW export with legally binding e-signatures that support Long-Term Validation (LTV), ensuring documents remain verifiable for years after signing.

**ETSI XAdES / CAdES**
- URL: https://www.esign.ai/blog/pades-vs-cades-vs-xades-explained
- Complementary ETSI signature standards: XAdES for XML-based SOW data, CAdES for binary file attachments. Less common in end-user SOW delivery but relevant for system-to-system exchange of structured contract data.

**RFC 3161 — Time-Stamping Protocol**
- URL: https://datatracker.ietf.org/doc/html/rfc3161
- Defines a cryptographic time-stamping protocol providing proof that a document's hash existed at or before a specific time. Important for long-lived SOWs where timestamp integrity must be demonstrable long after issuance.

**GDPR (EU) 2016/679**
- URL: https://gdpr-info.eu/
- Applies when the SOW generator processes personal data (client contact details, consultant profiles, rate card information tied to individuals). Requires Data Protection Impact Assessments (DPIA) for AI-driven processing, data minimisation, the right to erasure, and lawful basis for processing. Data stored for training must be anonymised.

### MCP Server Specifications

The Model Context Protocol (MCP) is directly relevant if the SOW generator exposes its document generation, clause library, and rate card lookup capabilities as MCP tools for AI agent consumers (e.g. sales copilots, project management assistants).

- **MCP Specification**: https://modelcontextprotocol.io/
- Key tool categories to expose via MCP: `generate_sow_draft`, `lookup_rate_card`, `retrieve_past_project`, `list_clause_templates`, `submit_for_approval`
- MCP resources: SOW documents, clause libraries, project archives, client profiles
- Authentication: OAuth 2.0 or API key, following MCP's authorisation guidance

## Notes

**Emerging standards gap — SOW data model:** There is no published industry standard (ISO, ANSI, or otherwise) defining a canonical data model for Scope of Work documents. Most implementations use proprietary schemas. An open-source SOW generator has an opportunity to publish a JSON Schema definition for a canonical SOW data model, which could become a de-facto community standard.

**Procurement-specific standards:** Government and enterprise procurement contexts (US FAR/DFARS, EU procurement directives) impose specific format requirements for SOWs and Statements of Objectives (SOO). These are regulatory rather than technical standards and are not captured by any single ISO or IETF specification. Compliance with these frameworks is a meaningful differentiator for tools targeting public-sector buyers.

**AI Act (EU) 2024/1689:** As of 2026, the EU AI Act applies to AI systems used in professional services workflows. A SOW generator using AI to draft legally relevant documents may be classified as a limited-risk system (subject to transparency obligations) or, if used in regulated procurement, as a high-risk system (subject to conformity assessment). This is an emerging compliance consideration with limited tooling currently available.
